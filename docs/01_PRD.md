# Teacher Maker PRD v1.0

> 교사를 위한 30분 바이브 코딩 입문 웹앱  
> 개발을 몰라도 AI와 함께 나만의 첫 번째 웹앱을 만들어보는 공개형 학습 사이트

---

## 0. 문서 정보

- 문서명: Teacher Maker PRD
- 버전: v1.0
- 작성 목적: Codex 또는 AI 개발 도구가 본 문서를 기반으로 웹앱을 구현할 수 있도록 제품 목표, UX, 화면 구조, 콘텐츠, 기술 구조를 명확히 정의한다.
- 프로젝트 유형: 정적 웹앱 / 공개 링크용 학습 사이트
- 1차 배포 대상: 교내 연수 참가자 및 바이브 코딩을 처음 접하는 교사
- 핵심 도구: Canva AI, Google AI Studio, ChatGPT
- 향후 확장 도구: GitHub, Codex, Vercel, Markdown 기반 콘텐츠 관리

## 1. 프로젝트 개요

### 1.1 프로젝트명

**Teacher Maker**

### 1.2 한 줄 설명

개발 경험이 없는 교사가 30분 안에 AI와 함께 첫 번째 웹앱을 만들어보는 바이브 코딩 입문 사이트.

### 1.3 핵심 메시지

> “개발을 몰라도 괜찮습니다.  
> 30분이면 AI와 함께 나만의 첫 번째 웹앱을 만들 수 있습니다.”

### 1.4 프로젝트 배경

많은 교사들은 AI와 코딩에 관심은 있지만, 실제로 시작하기 전 다음과 같은 장벽을 느낀다.

- 코딩이라는 말 자체가 어렵게 느껴진다.
- 어떤 도구부터 써야 할지 모른다.
- 개발 환경 세팅, GitHub, 배포 같은 용어가 부담스럽다.
- AI에게 무엇을 어떻게 요청해야 하는지 모른다.
- 완성 경험 없이 설명만 듣고 끝나는 경우가 많다.

이 프로젝트는 “바이브 코딩을 이론으로 배우는 사이트”가 아니다.  
교사가 실제 자신의 불편함을 떠올리고, AI에게 설명하고, 30분 안에 작고 단순한 웹앱을 만들어보게 하는 실습형 안내 사이트다.

### 1.5 제품의 핵심 관점

이 웹앱의 목표는 사용자가 “바이브 코딩을 이해했다”가 아니라 다음 생각을 하게 만드는 것이다.

> “나도 하나 만들어볼까?”

---

## 2. 제품 목표

### 2.1 Primary Goal

교내 연수 참가자 및 초보 교사가 웹사이트를 따라가며 30분 안에 첫 번째 AI 웹앱 제작 과정을 경험하도록 돕는다.

### 2.2 Secondary Goals

- 교사가 자신의 업무와 수업에서 해결할 수 있는 작은 문제를 발견하게 한다.
- 좋은 AI 요청문과 나쁜 AI 요청문의 차이를 직관적으로 이해하게 한다.
- Canva AI 또는 Google AI Studio를 활용해 웹앱 제작 진입 장벽을 낮춘다.
- 첫 성공 경험 이후 GitHub, Codex, 배포, 유지보수로 자연스럽게 확장할 수 있는 학습 경로를 제시한다.
- 연수 이후에도 공개 링크로 다시 방문해 복습할 수 있게 한다.

### 2.3 성공 지표

MVP 기준 성공 지표는 다음과 같다.

- 사용자가 “시작하기” 버튼을 눌러 30분 로드맵 페이지에 진입한다.
- 사용자가 최소 1개의 실습 프로젝트 프롬프트를 복사한다.
- 사용자가 단계별 체크리스트 중 5개 이상을 완료한다.
- 사용자가 실습 종료 후 “나도 하나 만들어볼까?”라는 감각을 얻는다.
- 연수자가 별도 설명 없이 사이트만 보고도 흐름을 따라갈 수 있다.

## 3. 타깃 사용자

### 3.1 Primary Persona: 바이브 코딩을 처음 배우는 교사

- 개발 경험 없음
- ChatGPT는 사용해본 적이 있을 수 있음
- Canva, Google 도구에는 비교적 익숙함
- “코딩”이라는 단어에 부담이 있음
- 짧은 시간 안에 실용적인 결과물을 원함
- 학교 업무와 학급 운영에 바로 쓸 수 있는 예시를 선호함

#### 사용자 니즈

- 쉽게 시작하고 싶다.
- 복잡한 세팅 없이 따라 하고 싶다.
- 내 업무에 도움이 되는 예시를 보고 싶다.
- 실습 중 막히면 바로 쓸 수 있는 문장을 원한다.
- 한 번 성공한 뒤 다음에 무엇을 하면 좋을지 알고 싶다.

### 3.2 Secondary Persona: 교내 연수 참가자

- 연수 시간 내 실습을 따라가야 함
- 강사의 안내와 함께 웹사이트를 보조 자료로 사용
- 연수 이후에도 다시 접속해 복습할 가능성이 있음
- 동료 교사에게 공유할 수 있는 공개 링크를 원함

### 3.3 Excluded Persona

MVP에서는 다음 사용자를 주요 대상으로 삼지 않는다.

- 이미 React, Next.js, GitHub, Codex를 능숙하게 사용하는 사용자
- 개발자 수준의 세부 기술 설명을 원하는 사용자
- 로그인 기반 학습 관리가 필요한 기관 관리자

## 4. 핵심 UX 철학

### 4.1 설명보다 경험

사용자는 긴 설명을 읽기보다 작은 성공 경험을 먼저 해야 한다.  
따라서 모든 화면은 짧은 설명, 명확한 예시, 바로 실행 가능한 복사 버튼 중심으로 설계한다.

### 4.2 30분 안에 성공 경험

MVP의 모든 흐름은 30분 연수 안에서 완료 가능해야 한다.  
긴 학습 로드맵보다 “오늘 하나 만들어보는 경험”에 집중한다.

### 4.3 어려운 용어 최소화

MVP 초반에는 다음 용어를 되도록 사용하지 않는다.

- API
- Framework
- Frontend
- Backend
- Repository
- Deploy
- Package
- Component
- State

필요한 경우 쉬운 말로 바꾼다.

예:

- Deploy → 공유하기
- Repository → 프로젝트 보관함
- Frontend → 화면
- Backend → 데이터를 저장하고 처리하는 뒷부분

### 4.4 복붙 중심

초보자는 빈 화면에서 시작하면 막힌다.  
따라서 모든 핵심 단계에는 “복사하기” 버튼이 있어야 한다.

### 4.5 교사의 실제 상황 중심

예시는 개발 학습용 예제가 아니라 교사 업무와 학급 운영에서 바로 떠올릴 수 있는 사례여야 한다.

예:

- 우리반 랜덤 뽑기
- 오늘 저녁 뭐 먹지
- 자리 바꾸기
- 인증장 문구 생성기
- 학급 퀴즈
- 급식 메뉴 기반 저녁 추천

MVP에서는 실습 프로젝트를 1~2개로 제한한다.

## 6. 정보 구조 IA

### 6.1 최상위 메뉴

```text
Home
├── 바이브 코딩이란?
├── 30분 로드맵
├── 실습 프로젝트
├── 프롬프트 라이브러리
├── FAQ
└── 다음 도전
```

### 6.2 권장 URL 구조

```text
/
 /intro
 /roadmap
 /projects
 /projects/random-picker
 /projects/dinner-menu
 /prompts
 /faq
 /next
```

MVP에서는 단일 페이지 스크롤 구조로 구현해도 된다.  
다만 향후 확장을 위해 섹션 ID 또는 라우팅이 가능한 구조로 설계한다.

### 6.3 우선 구현 방식

MVP는 다음 중 하나로 구현한다.

#### 권장안 A: 단일 페이지 랜딩형

- 첫 배포가 빠름
- 연수 상황에서 사용하기 쉬움
- 사용자가 위에서 아래로 자연스럽게 따라감

#### 확장안 B: 라우트 분리형

- 프로젝트가 늘어날 때 유리
- 각 프로젝트별 공유 링크 제공 가능
- SEO에 유리

최초 구현은 A로 시작하되, 컴포넌트는 B로 확장 가능하게 작성한다.

## 8. 화면별 상세 명세

## 8.1 Home 화면

### 목적

첫 방문자가 사이트의 목적을 5초 안에 이해하고, 부담 없이 시작 버튼을 누르게 한다.

### 핵심 카피

```text
개발을 몰라도 괜찮습니다.

30분이면
AI와 함께 나만의 첫 번째 웹앱을 만들 수 있습니다.
```

### 보조 카피

```text
Teacher Maker는 교사를 위한 바이브 코딩 입문 가이드입니다.
어려운 개발 용어보다, 실제로 하나 만들어보는 경험에 집중합니다.
```

### CTA 버튼

- Primary: `30분 로드맵 시작하기`
- Secondary: `실습 프로젝트 보기`

### 하단 카드

4개의 장점 카드 표시

1. 교사를 위한 예제
2. 복붙 중심 실습
3. 무료 AI 도구 활용
4. 첫 웹앱 완성 경험

### 디자인

- 넓은 여백
- 큰 타이포그래피
- 중앙 정렬 Hero
- 카드형 요약
- Notion/Apple 스타일의 차분한 디자인

### Acceptance Criteria

- 사용자는 5초 안에 이 사이트가 무엇인지 이해할 수 있다.
- Primary CTA가 화면 첫 영역에서 명확히 보인다.
- PC 화면에서 Hero 영역이 답답하지 않게 보인다.

## 8.2 바이브 코딩 소개 섹션

### 목적

바이브 코딩을 어려운 개발 방식이 아니라 “AI와 함께 문제를 해결하는 흐름”으로 이해하게 한다.

### 핵심 구조

```text
불편함 발견
↓
아이디어 떠올리기
↓
AI에게 설명하기
↓
웹앱 만들기
↓
사용하며 고치기
```

### 설명 문구

```text
바이브 코딩은 코드를 처음부터 직접 쓰는 것이 아닙니다.
내가 만들고 싶은 것을 AI에게 설명하고,
AI가 만든 결과를 보며 계속 대화하면서 개선해가는 방식입니다.
```

### 교사용 예시

```text
예전에는 “학생 랜덤 뽑기 도구가 있으면 좋겠다”에서 끝났다면,
이제는 AI에게 설명해서 직접 작은 웹앱으로 만들어볼 수 있습니다.
```

### 시각 요소

- 5단계 흐름 카드
- 화살표 또는 세로 타임라인
- 각 단계에 아이콘 사용 가능

### Acceptance Criteria

- 개발 경험이 없는 교사도 바이브 코딩의 전체 흐름을 이해할 수 있다.
- API, 프레임워크 등 어려운 용어가 등장하지 않는다.

## 8.3 30분 로드맵 섹션

### 목적

사용자가 실제 실습을 따라가며 첫 웹앱 제작 과정을 경험하게 한다.

### 구성

총 7단계로 구성한다.

```text
STEP 1. 불편함 찾기
STEP 2. 만들고 싶은 것 말로 설명하기
STEP 3. AI에게 줄 요청문 만들기
STEP 4. Canva AI 또는 Google AI Studio 열기
STEP 5. 붙여넣고 생성하기
STEP 6. 수정 요청하기
STEP 7. 완성하고 공유하기
```

### 공통 UI

각 단계 카드는 다음 요소를 포함한다.

- 단계 번호
- 제목
- 한 줄 설명
- 체크리스트
- 예시 문장
- 복사 버튼이 필요한 경우 CopyButton 표시
- 완료 체크박스

### STEP 1. 불편함 찾기

#### 설명

```text
먼저 학교생활이나 학급 운영에서 반복되거나 귀찮은 일을 하나 떠올려봅니다.
처음에는 아주 작은 문제일수록 좋습니다.
```

#### 체크리스트

- 반복해서 하는 일인가?
- 시간이 조금 아깝다고 느꼈던 일인가?
- 학생이나 교사가 버튼 하나로 쉽게 해결할 수 있는 일인가?
- 결과가 단순해도 괜찮은가?

#### 예시

```text
학생을 랜덤으로 뽑고 싶다.
오늘 저녁 메뉴를 추천받고 싶다.
학급 퀴즈를 쉽게 만들고 싶다.
```

### STEP 2. 만들고 싶은 것 말로 설명하기

#### 설명

```text
좋은 웹앱은 좋은 설명에서 시작합니다.
처음부터 완벽하게 쓸 필요는 없습니다.
누가, 언제, 무엇을 하려고 쓰는지만 적어도 충분합니다.
```

#### 좋은 설명 구조

```text
누가 사용할지
무엇을 하고 싶은지
꼭 필요한 기능
보기 좋았으면 하는 방식
```

#### 예시

```text
초등교사가 사용하는 랜덤 뽑기 웹앱을 만들고 싶다.
학생 이름을 입력해두고 버튼을 누르면 한 명이 랜덤으로 뽑히면 좋겠다.
화면은 교실 전자칠판에서 보기 좋게 크고 단순했으면 좋겠다.
```

### STEP 3. AI에게 줄 요청문 만들기

#### 설명

```text
AI에게는 짧지만 구체적으로 요청해야 합니다.
“웹앱 만들어줘”보다 “누가, 어떤 상황에서, 어떤 기능을 쓰는지”를 알려주는 것이 좋습니다.
```

#### 나쁜 예

```text
웹앱 만들어줘.
```

#### 좋은 예

```text
초등교사가 교실에서 사용할 수 있는 랜덤 뽑기 웹앱을 만들어줘.
학생 이름을 여러 명 입력할 수 있고,
버튼을 누르면 한 명이 랜덤으로 뽑히게 해줘.
전자칠판에서 보기 좋게 버튼과 글씨를 크게 만들어줘.
디자인은 깔끔하고 따뜻한 느낌이면 좋겠어.
```

### STEP 4. Canva AI 또는 Google AI Studio 열기

#### 설명

```text
처음에는 복잡한 개발 도구보다 바로 결과를 볼 수 있는 AI 도구를 사용하는 것이 좋습니다.
Canva AI 또는 Google AI Studio에 요청문을 붙여넣고 결과를 확인합니다.
```

#### 버튼

- `Canva 열기`
- `Google AI Studio 열기`

실제 링크는 프로젝트 구현 시 환경에 맞게 설정한다.

### STEP 5. 붙여넣고 생성하기

#### 설명

```text
복사한 요청문을 붙여넣고 AI가 만든 결과를 확인합니다.
처음 결과가 완벽하지 않아도 괜찮습니다.
바이브 코딩은 한 번에 완성하는 것이 아니라 보면서 고쳐가는 과정입니다.
```

### STEP 6. 수정 요청하기

#### 설명

```text
결과를 보고 마음에 들지 않는 부분을 말로 다시 요청합니다.
이 과정이 바이브 코딩에서 가장 중요합니다.
```

#### 수정 프롬프트 예시

```text
버튼을 더 크게 만들어줘.
전자칠판에서 잘 보이도록 글씨를 키워줘.
전체 디자인을 더 깔끔하게 바꿔줘.
학생 이름을 수정할 수 있게 해줘.
뽑힌 학생 이름이 더 크게 나오게 해줘.
```

### STEP 7. 완성하고 공유하기

#### 설명

```text
작아도 괜찮습니다.
오늘의 목표는 완벽한 서비스가 아니라 “내가 AI와 함께 하나 만들었다”는 경험입니다.
```

#### 완료 메시지

```text
🎉 첫 번째 AI 웹앱 완성!
이제 두 번째 웹앱은 훨씬 쉬워집니다.
```

## 8.4 실습 프로젝트 섹션

### 프로젝트 1: 우리반 랜덤 뽑기

#### 난이도

⭐ 쉬움

#### 예상 시간

15분

#### 추천 대상

- 학급에서 발표자, 도우미, 순서를 랜덤으로 정하고 싶은 교사
- 첫 번째 웹앱을 아주 단순하게 만들어보고 싶은 교사

#### 배울 수 있는 것

- 버튼
- 학생 이름 목록
- 랜덤 선택
- 큰 화면 UI
- 간단한 수정 요청

#### 기본 프롬프트

```text
초등교사가 교실에서 사용할 수 있는 “우리반 랜덤 뽑기” 웹앱을 만들어줘.

조건은 다음과 같아.

1. 학생 이름을 여러 명 입력할 수 있어야 해.
2. [뽑기] 버튼을 누르면 입력된 이름 중 한 명이 랜덤으로 선택되어야 해.
3. 뽑힌 학생 이름은 화면 가운데 아주 크게 보여줘.
4. 전자칠판에서 사용할 예정이라 글씨와 버튼을 크게 만들어줘.
5. 디자인은 깔끔하고 따뜻한 교실 분위기로 만들어줘.
6. 코딩을 모르는 교사도 사용할 수 있을 만큼 화면 구성을 단순하게 해줘.
7. 가능하면 하나의 HTML 파일로 실행할 수 있게 만들어줘.
```

#### 수정 프롬프트

```text
뽑힌 학생 이름이 더 크게 보이게 해줘.
버튼 색을 더 밝고 따뜻하게 바꿔줘.
학생 이름을 입력하는 칸을 더 넓게 만들어줘.
한 번 뽑힌 학생은 잠시 제외하는 기능을 추가해줘.
```

#### 확장 아이디어

- 모둠 뽑기
- 발표 순서 정하기
- 청소 구역 랜덤 배정
- 도우미 랜덤 배정

---

### 프로젝트 2: 오늘 저녁 뭐 먹지

#### 난이도

⭐⭐ 쉬움~보통

#### 예상 시간

20분

#### 추천 대상

- 메뉴 추천처럼 조건에 따라 결과를 보여주는 웹앱을 만들어보고 싶은 교사
- 카드형 UI를 경험해보고 싶은 사용자

#### 배울 수 있는 것

- 조건 선택
- 추천 결과
- 카드 UI
- 간단한 필터

#### 기본 프롬프트

```text
“오늘 저녁 뭐 먹지?” 웹앱을 만들어줘.

사용자는 집에서 저녁 메뉴를 고르기 어려울 때 이 웹앱을 사용해.

조건은 다음과 같아.

1. 사용자가 원하는 분위기를 선택할 수 있게 해줘.
   예: 간단한 메뉴, 든든한 메뉴, 가벼운 메뉴
2. [추천받기] 버튼을 누르면 저녁 메뉴 하나를 추천해줘.
3. 추천 결과에는 메뉴명, 추천 이유, 간단한 조리 아이디어를 보여줘.
4. 결과는 카드 형태로 깔끔하게 보여줘.
5. 전체 디자인은 Notion처럼 깔끔하고 편안한 느낌이면 좋겠어.
6. 처음 사용하는 사람도 바로 이해할 수 있게 화면을 단순하게 만들어줘.
7. 가능하면 하나의 HTML 파일로 실행할 수 있게 만들어줘.
```

#### 수정 프롬프트

```text
메뉴 추천 결과를 3개씩 보여줘.
매운 음식은 제외하는 선택지를 추가해줘.
요리 시간이 20분 이하인 메뉴만 추천하는 버튼을 추가해줘.
카드 디자인을 더 깔끔하게 바꿔줘.
```

#### 확장 아이디어

- 가족 구성원별 메뉴 추천
- 냉장고 재료 기반 추천
- 급식 메뉴와 겹치지 않는 저녁 추천
- 주간 식단 생성

```text
나는 코딩을 잘 모르는 초등교사야.
```

```text
현재 웹앱의 기능은 유지하면서 디자인만 더 깔끔하게 바꿔줘.
```

```text
현재 웹앱에 다음 기능을 추가해줘.

[추가하고 싶은 기능]

기존 기능은 망가지지 않게 유지해줘.
처음 사용하는 교사도 쉽게 쓸 수 있도록 화면을 복잡하게 만들지 말아줘.
```

```text
이 웹앱이 처음 사용하는 교사에게 조금 어려워 보여.

사용 방법을 한 줄 안내로 추가해줘.
```

## 8.6 FAQ 섹션

### 목적

초보자가 실습 중 흔히 겪는 막힘을 해결한다.

### FAQ 목록

#### Q1. AI가 이상한 결과를 만들었어요.

A. 처음 결과가 이상한 것은 자연스러운 과정입니다.  
원하는 부분을 더 구체적으로 말해보세요.

추천 프롬프트:

```text
내가 원한 것은 더 단순한 화면이야.
기능은 줄이고, 버튼 하나와 결과 화면 중심으로 다시 만들어줘.
```

#### Q2. 화면이 너무 복잡해요.

추천 프롬프트:

```text
처음 사용하는 교사도 쉽게 사용할 수 있도록 화면을 더 단순하게 만들어줘.
중요하지 않은 설명은 줄이고,
가장 중요한 버튼과 결과만 크게 보여줘.
```

#### Q3. 글씨가 너무 작아요.

추천 프롬프트:

```text
교실 전자칠판에서 사용할 예정이야.
제목, 버튼, 결과 글씨를 전체적으로 더 크게 만들어줘.
멀리서도 잘 보이게 해줘.
```

#### Q4. 모바일에서도 보이게 하고 싶어요.

추천 프롬프트:

```text
이 웹앱을 스마트폰에서도 보기 좋게 바꿔줘.
화면 크기에 따라 버튼과 글씨가 자연스럽게 줄어들게 해줘.
```

#### Q5. 저장이 안 돼요.

A. 처음 만든 간단한 웹앱은 새로고침하면 입력한 내용이 사라질 수 있다.  
필요하면 LocalStorage 저장 기능을 추가하도록 요청한다.

추천 프롬프트:

```text
사용자가 입력한 내용이 새로고침 후에도 남아 있도록 저장 기능을 추가해줘.
브라우저에 저장되는 간단한 방식이면 충분해.
```

#### Q6. 이제 무엇을 더 하면 좋을까요?

A. 첫 번째 웹앱을 만든 뒤에는 실제 교실이나 생활에서 한 번 사용해보고, 불편한 점을 다시 수정하면 된다.

## 11. 기술 스택

### 11.1 권장 스택

- Next.js App Router
- TypeScript
- Tailwind CSS
- shadcn/ui
- Framer Motion

### 11.2 이유

#### Next.js

- 정적 사이트로 배포하기 쉬움
- 향후 페이지 확장과 SEO에 유리
- Vercel과 궁합이 좋음

#### TypeScript

- 컴포넌트 구조가 늘어날 때 안정적
- Codex가 리팩토링하기 쉬움

#### Tailwind CSS

- 빠르게 깔끔한 UI 구현 가능
- 디자인 시스템을 일관되게 적용하기 쉬움

#### shadcn/ui

- Button, Card, Accordion 등 기본 컴포넌트 품질 확보
- Notion 스타일과 잘 어울림

#### LocalStorage

- 로그인 없이 진도 저장 가능
- MVP에 적합

### 11.3 대안 스택

초기 구현을 더 단순하게 하고 싶다면 다음도 가능하다.

- Vite + React + TypeScript + Tailwind
- 단일 HTML/CSS/JS

다만 공개형 사이트로 확장할 계획이 있으므로 Next.js를 권장한다.

## 18. Codex 개발 지침

### 18.1 구현 원칙

Codex는 본 PRD를 기준으로 다음 원칙을 지켜 구현한다.

1. MVP 범위를 벗어나는 기능을 임의로 추가하지 않는다.
2. 로그인, 서버, DB는 구현하지 않는다.
3. 모든 진도 저장은 LocalStorage로 처리한다.
4. UI는 PC 중심으로 설계하되 기본적인 반응형은 지원한다.
5. 콘텐츠는 `src/data`에 분리한다.
6. 컴포넌트는 재사용 가능하게 나눈다.
7. 초보 교사에게 어려운 문구를 UI에 넣지 않는다.
8. 복사 버튼은 모든 프롬프트 카드에 제공한다.
9. 접근성과 가독성을 우선한다.
10. 배포 가능한 상태까지 구현한다.

### 18.2 첫 개발 목표

첫 번째 개발에서는 다음을 완성한다.

- Next.js 프로젝트 초기화
- Tailwind 설정
- 기본 레이아웃
- Home Hero
- Intro Section
- Roadmap Section
- Project Section
- Prompt Library Section
- FAQ Section
- Next Challenge Section
- CopyButton
- Progress 체크 및 LocalStorage 저장
- 기본 SEO 메타데이터

### 18.3 개발 제외 사항

첫 버전에서 다음은 구현하지 않는다.

- 관리자 페이지
- Markdown 자동 로딩
- 검색 기능
- 다크 모드
- 사용자 계정
- 서버 저장
- 외부 API 연동
- 실제 AI 생성 기능 내장

## 19. Acceptance Criteria

### 19.1 기능 기준

- 사용자는 30분 로드맵의 각 단계를 완료 체크할 수 있다.
- 완료 상태는 새로고침 후에도 유지된다.
- 모든 주요 프롬프트는 버튼으로 복사할 수 있다.
- 복사 성공 여부가 사용자에게 표시된다.
- 실습 프로젝트 2개가 카드 형태로 제공된다.
- FAQ는 펼치고 접을 수 있다.
- 사이트는 로그인 없이 사용할 수 있다.
- 공개 링크로 접근 가능해야 한다.

### 19.2 UX 기준

- 첫 방문자는 5초 안에 사이트 목적을 이해할 수 있다.
- 사용자는 첫 화면에서 바로 시작 버튼을 찾을 수 있다.
- 초보 교사가 개발 용어 때문에 막히지 않아야 한다.
- 연수 중 강사가 별도 자료 없이도 사이트 흐름에 따라 안내할 수 있어야 한다.
- 복사해서 붙여넣는 흐름이 명확해야 한다.

### 19.3 디자인 기준

- 전체 디자인은 깔끔하고 차분해야 한다.
- 카드 간 여백이 충분해야 한다.
- 글씨가 작지 않아야 한다.
- PC 화면에서 연수 자료처럼 보기 좋아야 한다.
- 과한 색상과 장식을 피한다.

### 19.4 기술 기준

- TypeScript 에러가 없어야 한다.
- 빌드가 성공해야 한다.
- LocalStorage 접근 시 클라이언트 환경을 고려해야 한다.
- Lighthouse 성능 점수 90 이상을 목표로 한다.
- 배포 가능한 README를 포함한다.

## 22. 최종 요약

Teacher Maker는 교사에게 코딩을 가르치는 사이트가 아니다.

이 사이트의 핵심은 다음 한 문장이다.

> “내가 필요한 것을 AI와 함께 직접 만들어볼 수 있다.”

MVP는 복잡한 기능보다 첫 성공 경험에 집중한다.  
사용자가 30분 안에 작은 웹앱 하나를 만들고, 그 경험을 바탕으로 다음 도전을 떠올리게 하는 것이 가장 중요하다.

따라서 개발자는 다음을 항상 우선해야 한다.

1. 쉽게 이해되는가?
2. 바로 따라 할 수 있는가?
3. 복사해서 쓸 수 있는가?
4. 교사에게 실제로 의미 있는가?
5. 첫 성공 경험을 주는가?

이 기준을 만족한다면 Teacher Maker의 첫 버전은 성공이다.
